## 1. 项目概述

### 1.1 项目背景

在日常竞品调研、投资分析、市场研究等场景中，分析人员需要快速了解一个陌生网站的定位、商业模式和流量策略。传统方式需要人工浏览网站、搜索资料、对比竞品，耗时且依赖经验。

本项目开发一款 Chrome 浏览器扩展插件，用户在打开任意网页时，一键调用 AI 自动生成针对该网站的结构化分析报告，大幅提升网站调研效率。

### 1.2 项目目标

- **核心目标**：用户打开网页后，点击插件即可获得 AI 对该网站的三大维度分析
- **体验目标**：从点击到三问结果全部展示控制在 20 秒以内（并行调用）
- **MVP 定位**：本版为最小可用版本，聚焦"打开网页 → 一键分析 → 复制结果"核心闭环；暂不支持历史记录、PDF 导出等扩展能力
- **免费体验**：用户无需自备 API Key 即可体验，降低使用门槛

### 1.3 目标用户

| 用户画像 | 使用场景 |
|---------|---------|
| 产品经理 | 竞品调研、功能对标 |
| 投资人 / 分析师 | 项目尽调、商业模式评估 |
| 市场营销人员 | 流量策略研究、获客方式学习 |
| 创业者 | 行业摸底、寻找差异化机会 |

## 2. 功能需求

### 2.1 核心功能

插件需调用 AI 自动回答以下三个问题（以下简称"三问"）：

#### 问题一：网站定位与差异化
> 这个网站是干什么的？与主要的三个竞争对手有什么差异化的定位？

AI 需输出：
1. **网站概述**：用 1-2 段话说明网站是做什么的、服务谁、解决什么问题
2. **主要竞争对手**：列出 3 个核心竞争对手，并简述其特点
3. **差异化定位**：以对比表格形式呈现本网站与 3 个竞品在定位上的差异

#### 问题二：核心卖点与商业模式
> 这个网站的核心卖点是什么？怎么赚钱？

AI 需输出：
1. **核心卖点**：提炼 3-5 条核心卖点（USP）
2. **变现模式**：分析该网站的盈利方式（如广告、订阅、交易佣金、SaaS 等）
3. **定价策略**：如有可见的定价信息，总结其定价结构

#### 问题三：流量来源与用户留存
> 这个网站的流量来源是什么？如何留住客户？

AI 需输出：
1. **流量来源推断**：基于页面可见信息（SEO 关键词、社交分享按钮、广告位、内容策略等）推断主要流量来源
2. **获客方式**：分析其获客手段
3. **留存策略**：分析其如何留住用户（如会员体系、内容更新、推送通知等）

> **数据来源说明**：竞品及行业信息完全依赖 AI 模型自身知识，MVP 阶段不接入第三方数据 API（如 SimilarWeb）。AI 会在结果中标注信息为"基于模型知识推断"。

### 2.2 调用方式：并行调用

三问**并行**发起 3 个独立 AI 请求，以最大化响应速度：

- 一次分析同时发起 3 个请求（问题一/二/三）
- 哪个先返回就先渲染哪个，互不阻塞
- 三个结果全部返回后标记本次分析完成
- 任一请求失败时，该问支持单独重试，不影响其他两问

### 2.3 用户交互流程

```
用户打开目标网页
       │
       ▼
点击浏览器工具栏插件图标 ──► 弹出 Popup 窗口
       │
       ▼
点击「开始分析」按钮
       │
       ▼
插件提取当前页面内容（标题、描述、正文、元数据等）
       │
       ▼
并行发起 3 个 AI 请求（问题一/二/三）──► Popup 逐个渲染进度
       │
       ▼
结果以 Markdown 渲染展示在三个折叠面板中
       │
       ▼
用户可：复制单个结果 / 复制全部 / 重新分析
```

### 2.4 调用模式

插件支持两种调用模式，用户可在设置中切换：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **免费体验** | 走官方中转服务，无需 API Key，按设备限制每日次数 | 新用户试用、轻度使用 |
| **自有 Key** | 用户填入自己的 API Key，直连 AI 服务商，无次数限制 | 高频/重度用户 |

默认使用「免费体验」模式，首次打开即可用，降低上手门槛。

### 2.5 配置项

| 配置项 | 说明 | 默认值 |
|-------|------|-------|
| 调用模式 | 免费体验 / 自有 Key | 免费体验 |
| AI 服务商 | 选择 API 提供方（仅自有 Key 模式可选） | 通义千问 |
| API Key | 用户自有密钥（本地存储） | 空 |
| 模型选择 | 选择具体模型 | qwen-plus |
| 语言偏好 | 结果输出语言 | 跟随浏览器语言 |

**两条调用路径**：
- **免费体验**：插件 → 中转服务（托管共享 Key + 配额）→ AI 服务商
- **自有 Key**：插件 → AI 服务商（直连）

### 3.2 技术栈

| 模块 | 技术选型 | 说明 |
|------|---------|------|
| 扩展规范 | Manifest V3 | Chrome 当前标准，必选 |
| 前端框架 | 原生 JS / Vue 3 | Popup 较轻量，可选 |
| UI 样式 | CSS / Tailwind | 轻量美观 |
| 构建工具 | Vite | 打包扩展文件 |
| AI 接口 | Fetch API | 调用 AI 服务商 REST API |
| 状态存储 | chrome.storage.local | 存储 API Key 与配置 |
| 中转服务 | Cloudflare Worker / Vercel Edge Function | 免运维，托管共享 Key 与配额 |

### 3.3 模块划分

#### 3.3.1 Content Script（页面内容提取）
- **职责**：注入目标页面，提取分析所需的内容
- **提取内容**：
  - 页面标题（`document.title`）
  - meta 描述（`meta[name=description]`）
  - meta 关键词（`meta[name=keywords]`）
  - Open Graph 标签（`og:title`、`og:description`、`og:type`）
  - 主标题（h1、h2）
  - 页面正文文本（去噪后，限制长度避免超出 token）
  - 可见定价/会员信息（启发式提取）
  - 社交分享按钮、广告位、订阅入口等留存信号
- **通信**：通过 `chrome.runtime.sendMessage` 将提取内容发送给 Background

#### 3.3.2 Background Service Worker（AI 调用中枢）
- **职责**：接收 Content Script 数据，组织 Prompt，并行调用 AI API，返回结果
- **关键逻辑**：
  - 从 `chrome.storage` 读取调用模式与配置
  - 构建"三问"的 Prompt 模板
  - **并行**发起 3 个请求（`Promise.allSettled`），每问独立返回
  - 根据调用模式选择路径：免费体验 → 中转服务；自有 Key → 直连
  - 流式（SSE）可选，MVP 先用一次性返回以简化实现
  - 错误处理与单问重试（网络异常、Key 无效、超时、Token 超限）
- **通信**：与 Popup 通过 `chrome.runtime` 消息通信，每问返回即推送一次进度

#### 3.3.3 Popup UI（结果展示与设置）
- **职责**：提供用户交互界面
- **页面结构**：
  - 顶部：当前网站域名 + 「开始分析」按钮
  - 主体：三个折叠面板，分别对应三问结果
  - 底部：设置入口 + 复制按钮
- **加载态**：三问独立 loading，哪个先返回先渲染

## 4. 数据流设计

### 4.1 一次分析的完整时序（并行）

```
Popup          Background          Content Script        AI API / 中转
  │                │                    │                  │
  │ 1.点击分析     │                    │                  │
  │───────────────►│                    │                  │
  │                │ 2.发送提取指令      │                  │
  │                │───────────────────►│                  │
  │                │ 3.返回页面内容       │                  │
  │                │◄───────────────────│                  │
  │                │                    │                  │
  │                │ 4.并行发起3个请求                       │
  │                │──┬────────────────────────────────────►│ 问题一
  │                │   │─────────────────────────────────────►│ 问题二
  │                │   └─────────────────────────────────────►│ 问题三
  │                │                    │                  │
  │ 5.问题一返回   │                    │                  │
  │◄───────────────│                    │                  │
  │ 6.问题二返回   │                    │                  │
  │◄───────────────│                    │                  │
  │ 7.问题三返回   │                    │                  │
  │◄───────────────│                    │                  │
  │                │                    │                  │
  │ 8.三问全部完成  │                    │                  │
  │◄───────────────│                    │                  │
```

> 任一问失败不影响其他两问，失败项显示「重试」按钮。

### 4.2 Prompt 设计

为保证三问分析质量，每个 Prompt 需包含：
- **系统角色**：设定为资深互联网商业分析师
- **网站基本信息**：URL、标题、描述、关键词、正文摘要
- **任务指令**：明确输出结构（概述 + 竞品 + 差异化表格）
- **数据来源声明**：提示 AI 竞品信息基于模型自身知识，并要求在结果中标注推断性
- **输出格式**：要求 Markdown 格式，便于渲染

## 5. 非功能性需求

### 5.1 性能
- 页面内容提取应在 **2 秒**内完成
- 三问并行后，整体响应应在 **20 秒**内全部返回
- 提取的正文长度限制在 **8000 token** 以内，避免超出上下文

### 5.2 安全性
- 用户自有 API Key 仅存储在本地 `chrome.storage.local`，**不上传任何服务器**
- 中转服务仅转发请求，不持久化页面内容
- 所有 AI 请求通过 HTTPS
- 不收集、不上传用户浏览历史

### 5.3 兼容性
- 支持 Chrome 110+（Manifest V3 要求）
- 适配常见网站类型（电商、SaaS、内容站、企业官网）
- 对 SPA 网站兼容（等待 DOM 加载完成后再提取）

### 5.4 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| 当前页是 chrome:// 等内部页 | 提示"该页面无法分析" |
| 页面内容过少（如纯图片站） | 提示"内容不足，结果可能不准" |
| 免费体验额度用尽 | 提示今日次数已用完，引导切换自有 Key |
| 自有 Key 未配置 | 引导跳转设置页 |
| 自有 Key 无效 | 提示错误并引导重新配置 |
| 单问请求超时/失败 | 该问显示「重试」按钮，不影响其他两问 |
| Token 超限 | 自动截断正文后重试 |

### 5.5 免费体验配额策略

- 以设备为单位限制（首次使用生成匿名设备 ID，本地存储）
- 默认配额：每设备每日 **3 次完整分析**（即 9 次 AI 调用）
- 配额计数由中转服务侧判定，每日 0 点（UTC+8）重置
- 接近上限时在 Popup 提示剩余次数

## 9. 决策记录

以下决策已在需求评审中确认，作为本版基线：

1. **AI 服务商**：一期优先支持国内可直连服务商（通义千问、DeepSeek），MVP 默认通义千问。
2. **调用方式**：三问并行调用，优先速度；单问失败可独立重试。
3. **结果导出**：仅支持 Markdown 渲染 + 复制，**不支持 PDF 导出**。
4. **历史记录**：MVP 不支持保存历史，聚焦核心闭环快速交付。
5. **免费体验**：提供免费体验模式，用户无需自备 Key，通过中转服务托管共享 Key 并做设备配额限制。
6. **竞品数据来源**：竞品信息完全依赖 AI 模型自身知识，MVP 不接入第三方数据 API。
